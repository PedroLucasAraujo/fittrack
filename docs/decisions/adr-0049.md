# ADR-0049 — Toolchain Governance and Long-Term Code Integrity

## Status

ACCEPTED

## Context

FitTrack is a DDD modular monolith handling financial transactions (ADR-0004), health data (ADR-0037), tenant-isolated multi-professional billing (ADR-0025), and immutable execution records (ADR-0005). The domain invariants documented across ADR-0000 through ADR-0048 are enforced at the code level through TypeScript's type system, pure domain tests, and CI pipelines.

However, an audit of the current repository reveals that the toolchain itself is **not governed**:

| Gap | Risk |
|-----|------|
| No ESLint configuration | Domain layer isolation (ADR-0002) is not statically enforced. Nothing prevents a domain entity from importing an infrastructure adapter. Nothing prevents floating-point arithmetic on financial amounts (ADR-0004). |
| No Prettier configuration | Formatting drift across contributors. Review noise obscuring domain-critical changes. |
| `package-lock.json` in `.gitignore` | Every `npm install` resolves dependencies from scratch. Two developers on the same commit may get different dependency trees. CI and local environments diverge silently. This directly threatens financial determinism (ADR-0004) and execution immutability (ADR-0005) — a different Prisma version could alter query behavior. |
| All dependencies use caret (`^`) ranges | Combined with ignored lock files, the effective dependency set is unbounded. A minor-version bump in any dependency could introduce behavioral changes that violate domain invariants. |
| No Node.js version pinning in repository | Node 22 is specified only in CI. Local development may use any version. |
| Missing TypeScript strictness options | `noUncheckedIndexedAccess` is disabled, allowing silent `undefined` access on arrays and maps — dangerous for financial calculations. `exactOptionalPropertyTypes` is disabled, allowing `undefined` assignment to optional properties in DTOs. |
| No pre-commit quality gates | All enforcement is CI-only, providing feedback minutes after commit rather than seconds before. |

These gaps do not merely affect code style. They represent **structural weaknesses in the enforcement of domain invariants**. ADR-0045 (ADR Governance) establishes that ADRs are authoritative and code that contradicts an ADR is a bug. Without tooling enforcement, this principle relies entirely on human review discipline — which does not scale.

### Tooling as Architecture

In a DDD system with financial precision requirements, tenant isolation, and immutable records:
- **TypeScript strictness** is the first line of defense against type-level invariant violations.
- **Linting rules** are automated ADR compliance checks.
- **Deterministic dependency resolution** is a prerequisite for reproducible behavior in financial calculations.
- **Formatting** eliminates review noise, preserving reviewer attention for domain-critical changes.

Tooling governance is not cosmetic. It is architectural infrastructure.

## Decision

### §1. TypeScript Strictness Policy

All packages MUST use a shared base `tsconfig.base.json` at the repository root. Per-package `tsconfig.json` files extend it and specify only local overrides (`rootDir`, `outDir`, path references).

**Required compiler options (non-negotiable):**

| Option | Value | Rationale |
|--------|-------|-----------|
| `strict` | `true` | Enables all strict checks (already present). |
| `noUncheckedIndexedAccess` | `true` | Prevents silent `undefined` on array/map access. Critical for financial aggregations where accessing a missing index must be an explicit check, not a silent `undefined` that coerces to `NaN`. |
| `exactOptionalPropertyTypes` | `true` | Prevents `undefined` assignment to optional DTO properties. Forces explicit handling of "not provided" vs "provided as undefined". |
| `noFallthroughCasesInSwitch` | `true` | Prevents accidental fallthrough in status-machine switches (AccessGrant lifecycle, RiskStatus transitions). |
| `forceConsistentCasingInFileNames` | `true` | Prevents import resolution divergence between case-sensitive (Linux CI) and case-insensitive (Windows/macOS dev) file systems. |
| `noPropertyAccessFromIndexSignature` | `true` | Forces bracket notation for index signatures, making dynamic access explicit. |

**Options that remain as-is:**
- `target: ES2022` — aligned with Node 22 runtime capabilities.
- `module: ESNext` — ESM-first, consistent with `"type": "module"` in all packages.
- `moduleResolution: bundler` — appropriate for Vitest and future bundler integration.
- `skipLibCheck: true` — performance optimization; third-party type errors are not actionable.

### §2. Linting Policy

ESLint with flat config (`eslint.config.js`) at the repository root. All packages are linted from a single configuration.

**Rule categories and rationale:**

#### §2.1 Layer Isolation Enforcement
Restricted import patterns prevent domain-layer code from importing infrastructure or application-layer modules. This is the static equivalent of ADR-0002 (layered architecture) and ADR-0047 (aggregate root purity).

#### §2.2 Financial Safety
A custom restriction on `parseFloat`, `Number.parseFloat`, and known floating-point patterns in files under `domain/` and `application/` directories. All financial amounts MUST use integer cents (ADR-0004). The `Money` value object in `@fittrack/core` is the only sanctioned abstraction for monetary values.

#### §2.3 TypeScript Safety
- `@typescript-eslint/no-explicit-any` — `any` bypasses the type system entirely. In a domain where types enforce invariants, `any` is an escape hatch that must be flagged.
- `@typescript-eslint/no-non-null-assertion` — non-null assertions (`!`) bypass null checks. In a system with 5-check AccessGrant validation (ADR-0046), skipping null checks is dangerous.
- `@typescript-eslint/no-floating-promises` — unhandled promise rejections can silently swallow errors in domain event dispatch and notification delivery (ADR-0048).

#### §2.4 Scope
Linting applies to all `.ts` files. Test files (`__tests__/**`, `tests/**`) are linted but with relaxed rules where test ergonomics require it (e.g., `any` in test fixtures).

### §3. Formatting Policy

Prettier with minimal configuration at the repository root. The goal is deterministic formatting with zero debate.

| Option | Value | Rationale |
|--------|-------|-----------|
| `singleQuote` | `true` | Matches existing codebase convention. |
| `trailingComma` | `all` | Reduces diff noise on multi-line additions. |
| `printWidth` | `100` | Balances readability with modern wide displays. |
| `semi` | `true` | Explicit statement termination. Prevents ASI edge cases. |

Prettier and ESLint MUST NOT conflict. ESLint does not enforce formatting rules — Prettier owns all formatting.

### §4. Dependency Management Policy

#### §4.1 Version Pinning
All dependencies in `package.json` files MUST use exact versions (no `^`, no `~`). Exact versions ensure that `npm install` without a lock file produces the same direct dependency set.

#### §4.2 Lock File Commitment
`package-lock.json` MUST be committed to version control. It MUST NOT appear in `.gitignore`. The lock file is the sole deterministic record of the full resolved dependency tree. Without it, two installations of the same commit may produce different behavior — which is incompatible with financial determinism (ADR-0004) and execution immutability (ADR-0005).

#### §4.3 Upgrade Process
Dependency upgrades are performed via Dependabot PRs or manual `npm update`. Every upgrade:
- Runs the full CI pipeline (typecheck + test + lint).
- Is reviewed as a code change, not rubber-stamped.
- Production dependencies are upgraded individually. Dev dependencies may be grouped.

#### §4.4 Dependabot Configuration
Dependabot monitors all package directories on a weekly schedule. Production and development dependency PRs are separated. Major version bumps are not auto-merged — they require manual review. Open PR limit prevents Dependabot noise from overwhelming the review queue.

### §5. Node.js Version Pinning

A `.nvmrc` file at the repository root specifies the exact Node.js major version (`22`). This aligns local development with CI (which already specifies `node-version: "22"`). Contributors using `nvm` or `fnm` automatically switch to the correct version.

### §6. CI Integration

The CI pipeline is extended to include:
1. **Lint step**: `eslint .` at the repository root, running before tests.
2. **Format check**: `prettier --check .` at the repository root, running before tests.

Lint and format failures block the pipeline. No code merges to `main` without passing all quality gates.

### §7. Pre-Commit Enforcement

Quality gates run locally before commit via a pre-commit hook. The hook runs:
1. Prettier on staged files.
2. ESLint on staged files.

This provides sub-second feedback. CI remains the authoritative gate; pre-commit hooks are a convenience that prevents wasted CI cycles.

### §8. Upgrade and Amendment Policy

This ADR governs tooling configuration. Changes to tooling rules follow the same ADR amendment process defined in ADR-0045:
- Adding or removing an ESLint rule that enforces a domain invariant requires an ADR amendment.
- Changing TypeScript strictness options requires an ADR amendment.
- Changing dependency pinning strategy requires an ADR amendment.
- Formatting changes (Prettier options) do not require an ADR amendment — they are cosmetic.

## Invariants

1. `package-lock.json` is committed to version control. It is never listed in `.gitignore`.
2. All `package.json` dependencies use exact versions. Caret (`^`) and tilde (`~`) ranges are prohibited.
3. TypeScript `strict` mode and all options listed in §1 are enabled. Disabling any of them requires an ADR amendment.
4. ESLint is configured at the repository root and runs in CI. Lint failures block merges.
5. Domain-layer files never import from infrastructure or presentation layers. This is enforced by ESLint restricted import patterns.
6. All financial calculations use integer arithmetic through the `Money` value object. Floating-point functions are restricted in domain and application code.
7. The Node.js major version is pinned in `.nvmrc` and in CI configuration. They must match.

## Constraints

- ESLint rules that enforce domain invariants (layer isolation, financial safety) are not overridable per-file without explicit justification in a code review comment.
- Prettier formatting is not debatable. The configuration is minimal and deterministic. Contributors do not override Prettier locally.
- Dependabot PRs for major version bumps are not auto-merged. They require human review.
- Pre-commit hooks may be bypassed with `--no-verify` in emergencies, but CI still enforces all gates.

## Consequences

**Positive:**
- Domain invariants (layer isolation, financial precision, tenant scoping) gain static enforcement that does not depend on reviewer vigilance.
- Deterministic dependency resolution eliminates a class of "works on my machine" bugs that are especially dangerous in financial operations.
- Formatting consistency removes an entire category of review comments, preserving reviewer attention for domain-critical changes.
- New contributors receive immediate feedback on invariant violations before their code reaches CI.

**Negative:**
- Initial migration requires pinning all existing dependencies and committing lock files.
- `noUncheckedIndexedAccess` may require adding explicit `undefined` checks in existing code where array access was previously assumed safe.
- `exactOptionalPropertyTypes` may require updating DTO definitions to explicitly include `undefined` in union types.
- Pre-commit hooks add a small delay to the commit workflow.

## Dependencies

- ADR-0002: Layered Architecture (layer isolation enforced by ESLint)
- ADR-0004: Financial Precision (integer-cent enforcement by ESLint + TypeScript)
- ADR-0005: Execution Immutability (deterministic builds protect immutable record integrity)
- ADR-0010: Temporal Policy (TypeScript strictness prevents silent undefined in date computations)
- ADR-0025: Tenant Isolation (linting prevents cross-module imports that could bypass tenant scope)
- ADR-0037: Sensitive Data Handling (linting can flag known PII patterns in log statements)
- ADR-0045: ADR Governance (tooling changes follow ADR amendment process)
